# Review User Story

You are reviewing the user story specified by the user: $ARGUMENTS

## Instructions

Read the story file and provide a comprehensive review covering the following areas:

## 1. Completeness Check

Verify all sections are filled in:
- [ ] Frontmatter (id, priority, status, stack, feature, complexity)
- [ ] User Story (As a... I want... so that...)
- [ ] Business Outcome
- [ ] Technologies
- [ ] Acceptance Criteria (at least 5 specific criteria)
- [ ] Technical Specifications
- [ ] Testing Scenarios (Gherkin format)
- [ ] Resources
- [ ] Dependencies
- [ ] Notes
- [ ] Definition of Done

## 2. Clarity Review

Check that requirements are unambiguous:
- Is the user role clearly defined?
- Is the desired action specific?
- Is the business value clear?
- Are acceptance criteria measurable and verifiable?
- Are technical specs detailed enough to implement?

## 3. Testability Assessment

Evaluate if the story can be properly tested:
- Can each acceptance criterion be verified with a test?
- Do Gherkin scenarios cover happy path?
- Are edge cases and error scenarios included?
- Is the expected behavior clearly defined?

## 4. Security Review

Check for security considerations:
- [ ] Input validation requirements defined
- [ ] Authentication/authorization considered
- [ ] Data sanitization mentioned
- [ ] Rate limiting considered (if applicable)
- [ ] Sensitive data handling addressed
- [ ] OWASP Top 10 vulnerabilities considered

## 5. Accessibility Review

Check for accessibility requirements:
- [ ] WCAG 2.1 AA compliance mentioned
- [ ] Keyboard navigation considered
- [ ] Screen reader support noted
- [ ] Color contrast requirements
- [ ] Focus management addressed

## 6. Performance Considerations

Check for performance requirements:
- [ ] Response time expectations defined
- [ ] Data volume considerations
- [ ] Caching requirements (if applicable)
- [ ] Pagination needs (for lists)

## 7. Consistency Check

Verify alignment with project standards:
- Does it follow the story template format?
- Is the naming convention correct (story-XXX-slug.md)?
- Is it in the correct by-feature directory?
- Does it reference existing patterns and components?

## 8. Dependency Analysis

Check story dependencies:
- Are prerequisite stories identified?
- Are blocked stories noted?
- Are the dependencies reasonable?

## Review Output Format

Provide your review in this format:

### Summary
[Overall assessment: Ready / Needs Work / Major Issues]

### Strengths
- [What's good about this story]

### Issues Found
- [Specific issues with severity: Critical / Important / Minor]

### Suggestions
- [Specific improvements to make the story better]

### Questions for Clarification
- [Any ambiguities that need user input]
